# Agent 目录定义

**作者：Manus AI**  
**版本：v0.1**  
**状态：MVP 目录稿**

## 一、总体说明

本系统不应依赖一个“万能 Agent”承担所有职责，而应采用**多 Agent 分工**。这样做的原因很现实：项目管理、财务整理、会议纪要处理、知识库维护、新人培训与运维提醒的上下文差异很大，若全部塞进一个 Agent，容易出现权限过宽、提示词混乱、结果不稳定的问题。

因此，MVP 阶段建议采用“**一个总协调 Agent + 六个专用 Agent**”的结构。总协调 Agent 负责分发、汇总和提醒，专用 Agent 则聚焦于单一工作域。

## 二、Agent 总览

| Agent 名称 | 主要职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| `orchestrator_agent` | 总协调与路由 | Webhook、定时任务、人工触发 | 工作分派、汇总结果、异常升级 |
| `project_ops_agent` | 项目进度与任务管理 | 项目表、任务表、GitHub、会议行动项 | 待办更新、风险提示、周报摘要 |
| `finance_agent` | 财务整理与 U 折算 | 流水记录、汇率表、订阅表 | 收支台账、利润视图、异常提醒 |
| `meeting_agent` | 会议纪要与行动项抽取 | 录音转写、会议文档、聊天总结 | 纪要、行动项、任务候选 |
| `knowledge_agent` | 知识库整理与检索 | SOP、方案、纪要、项目文档 | 摘要、标签、知识索引 |
| `onboarding_agent` | 新人培训与考核 | 入职计划、SOP、任务反馈 | 学习路径、每日任务、评分记录 |
| `infra_agent` | 资产、部署、订阅与运维提醒 | 域名、服务器、站点、监控与账单 | 续费提醒、异常汇总、部署状态 |

## 三、各 Agent 详细定义

### 1. `orchestrator_agent`

`orchestrator_agent` 是系统入口，但不是最终决策者。它的职责是识别事件、分配到正确的专用 Agent、收集结果并统一发回 Telegram 控制群或后台面板。它不直接写高风险业务数据，除非由下游 Agent 返回结构化结果并通过规则校验。

| 项目 | 定义 |
| --- | --- |
| 触发方式 | Telegram 命令、Webhook、定时任务、后台按钮 |
| 读取范围 | 项目摘要、任务列表、Agent 状态、通知队列 |
| 写入范围 | `agent_runs`、`notifications`、汇总缓存 |
| 禁止动作 | 不得直接改财务金额、不得直接推生产环境变更 |

### 2. `project_ops_agent`

`project_ops_agent` 负责维持项目层面的可见性。它应定期读取项目、任务、会议行动项和 GitHub 变更，把这些内容整理成“下一步要做什么、谁在卡住、哪些里程碑有风险”的视图。对你这种多项目并行的公司来说，这是最先产生价值的 Agent。

| 项目 | 定义 |
| --- | --- |
| 输入 | `projects`、`tasks`、GitHub 提交、PR、Issue、会议行动项 |
| 输出 | 项目日报、项目周报、风险列表、逾期任务列表 |
| 关键能力 | 去重任务、识别阻塞、生成客户可读更新摘要 |
| 人工复核 | 客户正式发送前需人工确认 |

### 3. `finance_agent`

`finance_agent` 负责把多币种、多平台、多分类的收支整理成统一的 U 视图。它不应直接作为会计系统的唯一依据，但应成为经营管理的主视图。其重点是统一币种、识别分类、提示异常和汇总成本归属。

| 项目 | 定义 |
| --- | --- |
| 输入 | `finance_ledger`、`subscriptions`、`fx_rates`、账单邮件 |
| 输出 | 月度收支总览、项目利润估算、续费预警、异常流水提示 |
| 关键能力 | U 折算、周期费用归类、公司级与项目级成本拆分 |
| 人工复核 | 新增大额流水与汇率异常必须复核 |

### 4. `meeting_agent`

`meeting_agent` 用于把会议变成执行动作，而不是把会议记录留在文档里。它负责清洗转录文本、提炼结论、抽取行动项、推断责任人，并根据规则转成候选任务。

| 项目 | 定义 |
| --- | --- |
| 输入 | 会议录音转写、文档纪要、聊天整理 |
| 输出 | 会议摘要、行动项、任务候选、待确认风险点 |
| 关键能力 | 区分讨论与决策、标记待确认事项、生成标准纪要格式 |
| 人工复核 | 自动抽取的责任人与截止日期需复核 |

### 5. `knowledge_agent`

`knowledge_agent` 不是普通搜索器，而是公司记忆层的维护者。它负责收集项目文档、SOP、会议纪要、调研资料和交付方案，生成摘要、标签、关键词与索引信息，并写入向量检索层。

| 项目 | 定义 |
| --- | --- |
| 输入 | `knowledge_documents`、仓库文档、会议纪要、研究资料 |
| 输出 | 标签、摘要、检索片段、相关文章建议 |
| 关键能力 | 相似知识召回、文档去重、项目上下文串联 |
| 人工复核 | 对外输出的最终方案文档仍应复核 |

### 6. `onboarding_agent`

`onboarding_agent` 是给新人减压、给你减负的核心能力。它不只是发教程，而是根据 15 天考核路径，分配每日任务、检查进度、提醒遗漏、生成知识点清单，并记录学习成果与实际任务表现。

| 项目 | 定义 |
| --- | --- |
| 输入 | 培训计划、SOP、工具安装清单、任务反馈 |
| 输出 | 每日学习任务、阶段评分、卡点记录、补学建议 |
| 关键能力 | 分阶段引导、识别卡点、将训练与真实任务结合 |
| 人工复核 | 最终转正判断必须人工决定 |

### 7. `infra_agent`

`infra_agent` 负责公司与项目层面的运维资产可见性。它应该追踪域名、服务器、DNS、站点、订阅、监控和数据库备份策略，并把风险集中汇报，而不是等问题爆出来后再手工查。

| 项目 | 定义 |
| --- | --- |
| 输入 | `assets`、`subscriptions`、`deployments`、`project_databases`、监控事件 |
| 输出 | 续费提醒、停机提醒、部署状态摘要、备份缺失预警 |
| 关键能力 | 资产归属判断、即将到期提醒、生产风险聚合 |
| 人工复核 | 涉及生产环境修改时只给建议，不自动执行 |

## 四、Agent 之间的协作关系

多 Agent 系统的价值，不在于每个 Agent 都很聪明，而在于它们之间关系清晰。这里建议采用“总协调 + 专用处理 + 人工确认”的三级结构。所有外部输入先进入总协调 Agent，然后根据事件类型路由到对应 Agent。若输出内容涉及客户、财务、生产环境或正式报价，则统一进入人工确认队列。

| 来源事件 | 首个处理 Agent | 可能联动 Agent |
| --- | --- | --- |
| GitHub 提交 / PR | `project_ops_agent` | `knowledge_agent` |
| 新会议纪要 | `meeting_agent` | `project_ops_agent`、`knowledge_agent` |
| 新账单 / 支出 | `finance_agent` | `infra_agent` |
| 资产到期 / 监控告警 | `infra_agent` | `project_ops_agent` |
| 新人日报 / 培训提交 | `onboarding_agent` | `project_ops_agent` |
| 用户自然语言提问 | `orchestrator_agent` | 任意路由 |

## 五、权限边界

最容易失控的地方不是“Agent 不够强”，而是“Agent 权限太大”。因此必须把只读、建议写入、正式写入和外部执行区分开。

| 权限级别 | 含义 | 示例 |
| --- | --- | --- |
| 只读 | 仅分析，不改数据 | 读取项目与会议记录 |
| 建议写入 | 生成候选结果，待确认 | 生成任务、报价建议、财务分类 |
| 受控写入 | 满足规则后可写入主库 | 写入纪要摘要、通知、非敏感状态 |
| 外部执行 | 调用外部系统执行动作 | 发正式客户消息、修改生产配置 |

MVP 阶段，除非后续你明确授权并加入审批机制，否则所有 Agent 都**不应默认拥有外部执行权限**。

## 六、首批上线优先级

虽然上面列了七类 Agent，但不建议一开始全部做满。更现实的顺序是先让系统在项目、会议和新人培训三个方向见效，再逐步完善财务与运维层。

| 顺序 | Agent | 上线原因 |
| --- | --- | --- |
| 1 | `project_ops_agent` | 直接缓解你项目过多的压力 |
| 2 | `meeting_agent` | 让会议结果可执行 |
| 3 | `onboarding_agent` | 降低新人培养成本 |
| 4 | `finance_agent` | 统一经营视图 |
| 5 | `infra_agent` | 防止资产与订阅失控 |
| 6 | `knowledge_agent` | 提升检索与记忆能力 |
| 7 | `orchestrator_agent` | 作为成熟阶段统一入口 |

## 七、与 Skill 的结合方式

外部 Skill 不应直接等同于业务 Agent，但可以作为 Agent 的能力增强层。例如，`internal-comms` 可强化 `project_ops_agent` 的周报与更新写作，`meeting-insights-analyzer` 可强化 `meeting_agent` 的纪要解析，`git-pushing` 与 `using-git-worktrees` 可强化开发相关 Agent 的代码协作流程，`deep-research` 与 `content-research-writer` 则可增强 `knowledge_agent` 与方案整理流程。
