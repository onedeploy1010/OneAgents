# 产品范围说明

**作者：Manus AI**  
**版本：v0.1**  
**状态：MVP 定义稿**

## 一、产品目标

本系统旨在为一家以远程交付、AI 辅助开发和多项目并行为特征的小型技术公司，构建一个统一的“**运营操作系统**”。系统并不替代所有已有工具，而是把项目、财务、资产、人员培训、会议行动项和 Agent 自动化连接成一个可持续运行的中心层，使团队在项目数量增长、协作者增加之后，仍然能够保持信息清晰、责任清晰和交付可追踪。

系统的核心目标有四个。第一，是让项目进度、风险、待办事项和部署状态不再散落在聊天、代码平台和个人记忆中，而是回到一个可查询、可提醒、可归档的主系统。第二，是让财务信息从多币种、多来源、多人记录的混乱状态，收敛为统一以 **U** 为结算单位的运营账本。第三，是让会议纪要、调研结论、客户反馈和开发变更形成可检索的知识积累。第四，是让新人在 15 天内，在 AI 的辅助下快速进入可独立执行的工作状态。

## 二、产品面向的角色

本系统初期面向的不是大组织，而是小而快的交付团队。因此角色设计应简洁，但必须清楚。

| 角色 | 定义 | 核心目标 |
| --- | --- | --- |
| 创始人 / 主负责人 | 公司拥有者与总协调人 | 看到整体经营、项目、资金与风险全貌 |
| 协作开发者 | 能独立承担部分项目任务的人 | 接收任务、更新进度、提交交付 |
| 新人 / 试用成员 | 处于 15 天考核期的成员 | 在 SOP 与 Agent 引导下尽快完成上手 |
| Agent | 负责读取、整理、提醒、同步、建议的系统能力 | 降低手工整理与跟踪成本 |
| 客户 | 只接收交付状态、报价、里程碑结果的人 | 获得稳定透明的外部沟通体验 |

## 三、系统解决的问题

当前业务的主要问题不在于“没有工具”，而在于**事实分散**。项目状态在聊天里，开发信息在 GitHub，部署在 Cloudflare/Vercel/Netlify，数据在 Supabase，财务分散在多币种收支里，人员培训靠口头传递。随着客户增多，系统缺乏一个统一中枢，导致你本人承担了过多上下文管理工作。

因此，本系统要解决的问题不是单一模块，而是以下六类核心痛点。

| 问题类别 | 现状痛点 | 系统目标 |
| --- | --- | --- |
| 项目管理 | 多项目并行时容易漏任务、漏风险、漏回复 | 建立统一项目台账与任务状态流 |
| 财务整理 | USDT、USDC、USD、MYR、CNY 混杂，难比较真实利润 | 建立统一 U 记账与折算规则 |
| 会议转执行 | 纪要和行动项经常断裂 | 自动提取任务、责任人、截止时间 |
| 运维资产 | 域名、站点、服务器、订阅、数据库分散 | 建立统一资产与订阅账本 |
| 新人培养 | 新人反复问基础问题，训练成本高 | 用 SOP + Agent 实现自助上手 |
| 经营判断 | 报价与资源消耗缺少数据支撑 | 用项目历史与成本记录辅助报价 |

## 四、MVP 范围

MVP 的目标不是一次性做成完整 ERP，而是先跑通最关键的运营闭环。因此第一阶段只纳入那些直接影响项目交付、运营清晰度和培训效率的模块。

| 模块 | 是否纳入 MVP | 说明 |
| --- | --- | --- |
| 项目台账与任务看板 | 是 | 必须成为所有项目事实入口 |
| 客户与项目基础信息 | 是 | 包括客户、合同状态、项目平台、部署位置 |
| 会议纪要转任务 | 是 | 用于把沟通转为执行动作 |
| 财务流水与 U 折算 | 是 | 用于收入、支出与利润视图 |
| 资产与订阅管理 | 是 | 跟踪服务器、域名、邮箱、API、订阅 |
| 新人 15 天培训流程 | 是 | 支撑新成员快速入场 |
| Agent 日报 / 周报 / 风险提醒 | 是 | 让信息自动回流给你 |
| 客户消息自动回复 | 否 | 初期只做提醒与建议，不直接自动回复 |
| 自动报价引擎 | 否 | 初期做辅助建议，不做自动下单 |
| 全自动部署接管 | 否 | 初期保留人工确认 |

## 五、关键原则

这个系统必须遵守几个原则，否则随着团队增长很快会再次失控。

第一，**结构化事实优先**。任何项目、财务、资产、人员、订阅数据，都必须优先进入结构化主库，而不是只留在 Notion、聊天记录或个人脑中。第二，**Agent 负责整理和提醒，不直接越权执行高风险动作**。第三，**同一条信息只能有一个主事实源**，例如项目状态以主运营库为准、代码以 GitHub 为准、客户数据库以各项目 Supabase 为准。第四，**培训与交付都必须可追踪**，避免新人只是在“看起来忙”。

## 六、边界与不做事项

为了让项目能在现实中落地，必须明确本系统当前不做的事情。系统不取代 GitHub 做源码托管，不取代 Supabase 做客户业务数据库，不取代 Cloudflare 做边缘执行，不取代企业邮箱做正式邮件收发，也不在未授权情况下自动完成外部敏感操作，例如购买、付款、发正式客户消息或修改生产环境。

## 七、成功标准

如果 MVP 成功，应当出现几个可观察结果。第一，你不再需要手工追踪所有项目上下文。第二，新人能在 15 天内完成基础环境配置和简单任务交付。第三，每周能稳定看到项目进度、财务流水、异常风险和待处理事项汇总。第四，客户项目的资产、部署和数据库归属可被快速查询。第五，后续新增协作者时，系统不需要推倒重来。
