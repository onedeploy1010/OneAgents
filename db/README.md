# 数据库目录说明

**作者：Manus AI**  
**版本：v0.1**  
**状态：初版**

## 一、目录作用

本目录用于存放 **OneAgents 公司运营库** 的数据库迁移、种子数据与后续策略脚本。这里的数据库是公司自己的主运营库，不是客户业务数据库。客户项目的业务数据仍应保留在各自独立的 Supabase 项目中；本目录只负责公司内部项目、任务、会议、财务、资产、培训、通知与 Agent 日志等结构化事实。

## 二、当前文件

| 路径 | 作用 |
| --- | --- |
| `migrations/001_init.sql` | 初版主运营库结构，包含扩展、枚举、表、索引、触发器与基础视图 |
| `seeds/` | 后续放参考种子数据，例如默认汇率、示例分类、演示用户 |

## 三、推荐应用顺序

第一步，在 Supabase 新建**公司运营项目**，不要直接对客户项目库执行本目录脚本。第二步，在 SQL Editor 或迁移流程中执行 `migrations/001_init.sql`。第三步，验证核心表是否创建成功。第四步，再按业务需要补充种子数据、RLS 策略和服务角色写入约束。

| 顺序 | 动作 | 结果 |
| --- | --- | --- |
| 1 | 新建公司运营 Supabase 项目 | 获得独立主库 |
| 2 | 执行 `001_init.sql` | 建表成功 |
| 3 | 检查 `users`、`projects`、`tasks`、`finance_ledger` 等表 | 核心结构可用 |
| 4 | 再补种子与策略 | 进入真实开发阶段 |

## 四、首版注意事项

首版 SQL 的重点是**跑通主结构**，而不是一次性完成全部治理能力。因此当前脚本已经包含审计字段、基础枚举和几个最有价值的视图，但尚未完整加入 RLS、对象存储策略、Webhook 事件表与更细粒度审批流。后续建议优先补 `RLS`、`service role` 写入边界、以及财务异常复核策略。

## 五、与 Agent 的关系

Agent 不应直接把自然语言结果无条件写进所有业务表。更稳妥的做法是先把结果写入候选记录或待复核字段，再由人工确认。当前数据库中已经为 `tasks`、`meetings`、`meeting_action_items`、`finance_ledger` 等表预留了 `created_by_agent`、`agent_name`、`source_channel`、`confidence_score` 和 `needs_human_review` 等审计字段，供后续执行层使用。
