# 集成矩阵

**作者：Manus AI**  
**版本：v0.1**  
**状态：MVP 接入稿**

## 一、总体说明

本矩阵用于说明系统应连接哪些平台、连接的目标是什么、数据流向如何设计、应由谁持有主账号，以及是否属于 MVP 必接项。对于你当前阶段来说，这份矩阵的价值不只是“列清单”，而是建立一个统一原则：**任何接入都必须能回答接什么、读什么、写什么、谁负责、有什么风险**。

## 三、关键接入说明

### 1. GitHub

GitHub 是代码事实源，因此它不应该被运营系统覆盖，而应被运营系统解释和映射。系统要读取仓库、PR、Issue 和提交变化，并把这些变化映射回内部项目和任务。若未来要进行更深协作，可增加自动建 Issue 或自动生成变更摘要，但不建议初期让系统自动合并或直接操作生产分支。

| 维度 | 说明 |
| --- | --- |
| 主要用途 | 同步项目开发事实 |
| 读操作 | Repo、Branch、Issue、PR、Commit、Actions 结果 |
| 写操作 | 初期可不写；后续可写评论、Issue、标签 |
| 风险 | 自动写入过度会干扰开发流程 |

### 2. Supabase

Supabase 在这里要区分两层。第一层是**公司运营库**，它是运营系统的主事实库。第二层是**客户项目库**，它是客户业务系统的数据源。主运营库可以记录客户项目库的关键元数据、环境映射、备份状态和统计摘要，但不应把所有客户业务数据粗暴复制进来。

| 维度 | 公司运营库 | 客户项目库 |
| --- | --- | --- |
| 目标 | 保存公司经营与执行事实 | 保存客户项目业务事实 |
| 写入方式 | 系统直接写入 | 仅在授权下做映射、同步、备份 |
| 风险 | 权限过大需严格控制 | 客户数据隔离必须清晰 |

### 3. Cloudflare

Cloudflare 主要承担 Agent 入口、Webhook 接收、定时任务、轻量状态和边缘自动化。它非常适合做执行层，但不建议在当前阶段替代 Supabase 做公司所有主业务数据存储。

| 维度 | 说明 |
| --- | --- |
| 主要用途 | Worker、Agent、定时流程、通知编排 |
| 读取内容 | 外部事件、轻量配置、缓存状态 |
| 写入内容 | 通知、执行日志、临时状态 |
| 风险 | 若把核心经营数据全放边缘层，后期治理会变复杂 |

### 4. Telegram 与 WhatsApp

两个通讯平台的角色应该区分清楚。Telegram 更适合作为**内部控制台**，而 WhatsApp 更适合作为**客户沟通出口**。内部消息强调效率与结构化，客户消息强调审核、规范与品牌一致性。

| 平台 | 推荐定位 |
| --- | --- |
| Telegram | 团队内部命令、提醒、审批、日报 |
| WhatsApp | 客户通知、阶段汇报、模板消息、催款提醒 |

## 四、数据流方向建议

为了避免接入之后仍然混乱，所有集成应遵循统一数据流：**外部系统输入事件 → Cloudflare 执行层处理 → Supabase 主运营库存档 → Telegram / 面板 / 报表输出**。如果某个接入不符合这个结构，就说明它的角色还没有定义清楚。

| 来源 | 进入层 | 主存储 | 输出 |
| --- | --- | --- | --- |
| GitHub | Cloudflare Worker | Supabase | 面板、日报 |
| 会议纪要 | Cloudflare Worker | Supabase | 任务、知识库 |
| 邮件账单 | Cloudflare Worker | Supabase | 财务提醒 |
| 资产告警 | Cloudflare Worker | Supabase | Telegram 提醒 |
| 培训反馈 | Cloudflare Worker | Supabase | 评分、周报 |

## 五、账号归属与权限原则

接入的成败，很大程度取决于账号归属是否统一。所有关键平台都建议由**公司主组织账号**持有，而不是挂在个人私人邮箱下。成员通过团队权限进入；Agent 通过单独的服务账户或 API 凭证进入。客户级平台若由你代管，也必须在系统里记录归属关系、登录入口、续费责任和交接方式。

| 对象 | 建议归属 |
| --- | --- |
| GitHub Organization | 公司 |
| Supabase Organization | 公司 |
| Cloudflare Account | 公司 |
| 企业邮箱 | 公司域名 |
| Telegram Bot | 公司 |
| WhatsApp Business Account | 公司业务账号 |

## 六、首批落地顺序

虽然矩阵里列了多个平台，但真正的首批接入不需要太多。你现在最应该先接的五个，是 **GitHub、Supabase、Cloudflare、Google Workspace、Telegram**。这五项可以先把项目、任务、文档、提醒和培训跑起来。之后再根据客户沟通需求决定是否接 WhatsApp，根据项目部署方式再决定是否把 Vercel / Netlify 纳入更深集成。
