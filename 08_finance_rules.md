# 财务规则说明

**作者：Manus AI**  
**版本：v0.1**  
**状态：MVP 财务规则稿**

## 一、规则目标

本规则用于解决当前财务记录分散、多币种混用、项目与公司成本交叉不清的问题。规则的核心目标不是替代正式会计，而是建立一个适合技术交付团队的**经营视图**，让你能够快速知道每个项目赚了多少、每月固定成本是多少、哪些支出属于公司建设、哪些属于客户项目消耗。

本规则的中心思想是：**保留原始币种，统一折算成 U，按归属和类型进行分类，并对异常项进行人工复核。**

## 二、统一结算单位定义

系统内部统一使用 **U** 作为经营分析单位。这里的 U 不是用来替代真实支付币种，而是为了让 USDT、USDC、USD、MYR、CNY 等记录在同一张经营报表中可比较。换句话说，真实币种永远保留，U 只是统一分析单位。

| 字段 | 含义 |
| --- | --- |
| `original_currency` | 原始记账币种 |
| `original_amount` | 原始记账金额 |
| `rate_to_u` | 记账时使用的折算汇率 |
| `amount_u` | 折算后的统一分析金额 |

## 三、折算规则

折算规则必须稳定、可追溯，而不是随手估。对于稳定币如 USDT、USDC，若经营上认可其与 USD 基本等值，可按 1:1 近似计入 U；对于法币如 MYR、CNY，则必须有明确汇率来源或当日约定汇率。

| 币种 | 默认折算策略 |
| --- | --- |
| USDT | 默认 1.00 U |
| USDC | 默认 1.00 U |
| USD | 默认 1.00 U |
| MYR | 按当日约定汇率折算 |
| CNY | 按当日约定汇率折算 |
| 其他币种 | 必须手工确认后入账 |

### 折算原则

第一，若同一笔记录已发生真实兑换，则优先采用真实成交汇率。第二，若没有真实兑换记录，则采用系统约定汇率来源。第三，所有汇率必须记录时间点和来源。第四，涉及大额或争议金额时，必须人工确认后再入账。

## 四、收支分类规则

为了让经营分析有意义，所有流水都必须有分类，而且分类粒度必须足够支撑你做项目利润判断。

| 一级分类 | 二级示例 | 说明 |
| --- | --- | --- |
| 收入 | 项目首付款、尾款、月维护费、咨询费 | 进入公司收入 |
| 基础设施 | 服务器、对象存储、CDN、数据库、监控 | 技术运行成本 |
| 平台订阅 | GitHub、邮箱、设计工具、AI 工具、部署平台 | 公司运营工具成本 |
| 域名与站点 | 域名购买、续费、站点托管 | 可归属到客户或公司 |
| 人员成本 | 新人补贴、协作者报酬、奖金 | 团队成本 |
| 支付与手续费 | 提现费、换汇费、链上手续费、网关费 | 金融交易成本 |
| 其他 | 不属于以上分类的支出 | 必须备注 |

## 五、归属规则

分类之外，财务最重要的是“归属”。一笔支出如果归属错误，你看到的项目利润就会失真。因此每一笔流水必须至少明确是属于**公司级**还是**项目级**。

| 归属类型 | 定义 | 示例 |
| --- | --- | --- |
| 公司级 | 用于公司整体运转，不只服务单个项目 | 企业邮箱、GitHub Team、通用 AI 订阅 |
| 项目级 | 明确服务某个客户项目 | 某客户专用服务器、专用域名、专属 API |
| 共享级 | 多项目共享但可按比例分摊 | 主开发服务器、监控系统 |

### 分摊原则

共享级成本初期不必追求过度精细。MVP 阶段可以采用简单规则，例如按项目数量、按活跃项目、或按月工时比例分摊。等数据更稳定后，再升级为更精细的成本分摊模型。

## 六、收入确认规则

收入记录应尽量以“真实收到”为准，而不是以口头承诺为准。若客户已确认但尚未到账，可在系统中单独记录为“应收”，不要直接计入实际收入。对于分期项目，首付款、阶段款和尾款应分别入账，而不是合并处理。

| 状态 | 含义 |
| --- | --- |
| `expected` | 已确认但未收款 |
| `received` | 已到账 |
| `refunded` | 已退款 |
| `bad_debt_risk` | 存在坏账风险 |

## 七、周期费用规则

订阅类费用不应每个月靠人工想起来才记。凡是可预测的周期性支出，例如 GitHub、Google Workspace、Supabase、Cloudflare、Vercel、Notion 等，都应先在 `subscriptions` 表中登记，再自动或半自动同步到 `finance_ledger`。这样可以保证：第一，不漏记；第二，可提前提醒续费；第三，可稳定看到固定经营成本。

## 八、异常复核规则

财务模块的风险不在于少算一点，而在于把错误数据当真。因此以下情况必须进入人工复核队列。

| 情况 | 处理方式 |
| --- | --- |
| 大额支出超过阈值 | 必须复核 |
| 汇率与常规偏差过大 | 必须复核 |
| 同金额同时间重复入账 | 必须复核 |
| 无凭证的费用 | 必须复核 |
| 无项目归属但看似项目支出 | 必须复核 |
| Agent 自动分类置信度低 | 必须复核 |

## 九、月度财务视图建议

系统每月应至少生成三种经营视图。第一种是公司总览，看总收入、总支出、净 U。第二种是项目视图，看每个项目收入、项目直接成本和估算利润。第三种是固定成本视图，看公司每个月无论是否接单都必须承担的费用。

| 视图 | 目的 |
| --- | --- |
| 公司总览 | 判断整体经营健康度 |
| 项目利润视图 | 判断哪些项目值得继续做 |
| 固定成本视图 | 判断公司每月保底压力 |

## 十、建议的报价辅助逻辑

财务规则最终还应反哺报价。历史项目的 `quoted_u`、实际支出、维护周期和变更频率，应该逐步形成报价参考库。后续报价建议至少要考虑四类因素：开发量、维护量、客户沟通成本和外部资源成本。这样报价就不再完全依赖主观感觉。

## 十一、与正式会计的关系

本规则面向经营管理，不等于正式财务报税体系。正式报税、法币对账、公司法定报表等事务，仍应使用当地合规方式处理。本系统的职责是让你在经营层快速看懂业务，不是替代法律与税务流程。
